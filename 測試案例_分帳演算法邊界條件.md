## 4.1 待填金額（twd_pending）

### TC-PENDING-01：單筆 twd_pending 排除

**場景**：行程有 2 筆帳，其中 1 筆 twd_pending = true

**前置資料**：

| 筆 | 類型 | twd_amount | twd_pending | 付款人 | 參與 |
|----|------|-----------|-------------|--------|------|
| A | shared | 1200 | false | Ning | 全員(4) |
| B | shared | NULL | true | Ziyu | 全員(4) |

**執行**：觸發結算

**預期結果**：

- settlement_items 僅依筆 A 計算（B 完全排除）
- 每人 cost = 300（A 的均攤）
- Ning net_balance = 1200 − 300 = +900
- 其餘 3 人 net_balance = −300 each
- 結算頁顯示 ⚠️「仍有 1 筆台幣待填，結算結果可能不完整」
- 結算按鈕**可點擊**（不阻斷）

---

### TC-PENDING-02：全部 twd_pending

**場景**：行程所有帳目都 twd_pending = true

**前置資料**：

| 筆 | 類型 | twd_amount | twd_pending | 付款人 |
|----|------|-----------|-------------|--------|
| A | shared | NULL | true | Ning |
| B | individual | NULL | true | Xiu |

**執行**：觸發結算

**預期結果**：

- 所有人 payout = 0，cost = 0，net_balance = 0
- settlement_items = 空（無任何轉帳）
- 結算頁顯示 ⚠️「仍有 2 筆台幣待填，結算結果可能不完整」

---

### TC-PENDING-03：補填後重算覆蓋

**場景**：先結算（含 pending 筆），補填後重新結算

**流程**：

1. 狀態：筆 A（shared $1200，paid），筆 B（twd_pending = true）→ 執行結算 S1
2. 補填 B 的 twd_amount = 2000，twd_pending → false
3. trip.status 自動回退 active
4. 執行結算 S2

**預期結果**：

- S2 取代 S1 作為當前有效結算
- S1 保留於歷史（is_cleared 標記不影響 S2）
- S2 的 settlement_items 包含 A + B 的完整計算

## 4.2 split_pending（individual 模式部分成員待填）

### TC-SPLIT-01：單一成員 split_pending

**場景**：individual 費用，4 人參與，A 尚未填金額

**前置資料**：

| 成員 | split_amount | split_pending |
|------|-------------|--------------|
| Ning（付款人） | 2500 | false |
| Ziyu | 1800 | false |
| Xiu | NULL | true |
| Mei | 1000 | false |

twd_amount = 7000

**執行**：計算 cost

**預期結果**：

- Xiu 的 cost 貢獻 = 0（split_pending 排除）
- total_input = 2500 + 1800 + 1000 = 5300
- difference = 7000 − 5300 = 1700
- Ning effective_split = 2500 + 1700 = 4200
- Xiu 的 my_cost = 0（顯示 ⚠️ 含待填）

---

### TC-SPLIT-02：全員 split_pending

**場景**：individual 費用，所有參與成員 split_pending = true

**前置資料**：

| 成員 | split_amount | split_pending |
|------|-------------|--------------|
| Ning（付款人） | NULL | true |
| Ziyu | NULL | true |
| Xiu | NULL | true |

twd_amount = 3000

**執行**：計算 cost 與 net_balance

**預期結果**：

- 所有人 cost = 0
- difference = 3000 − 0 = 3000
- Ning effective_split = 0 + 3000 = 3000（但 split_pending = true，排除）
- 實際上整筆 individual cost 全為 0
- Ning payout = 3000，cost = 0，net_balance = +3000（偏高，待補填後重算）

---

### TC-SPLIT-03：補填後清除 split_pending 並重算

**流程**：

1. TC-SPLIT-01 情境：Xiu split_pending = true
2. Xiu 補填 split_amount = 1600，split_pending → false
3. 觸發重新結算

**預期結果**：

- difference = 7000 − (2500 + 1800 + 1600 + 1000) = 7000 − 6900 = 100
- Ning effective_split = 2500 + 100 = 2600
- 所有人 split_pending = false → 無 ⚠️ 警示

## 4.3 individual 模式差額歸付款人

### TC-DIFF-01：差額為正（各人加總 < twd_amount）

**場景**：免稅湊單，各人填的比實付少

**前置資料**：

twd_amount = 5000，付款人 = Ning（參與）

| 成員 | split_amount |
|------|-------------|
| Ning | 1000 |
| Ziyu | 1500 |
| Xiu | 1500 |

total_input = 4000，difference = +1000

**預期結果**：

- Ning effective_split = 1000 + 1000 = 2000
- Ning cost 中此筆計 2000
- 前端橘色警示：「差額 +$1000 歸付款人」

---

### TC-DIFF-02：差額為負（各人加總 > twd_amount）

**場景**：各人分別計算的金額加總超過實付（如各自四捨五入）

**前置資料**：

twd_amount = 5000，付款人 = Ning（參與）

| 成員 | split_amount |
|------|-------------|
| Ning | 1800 |
| Ziyu | 1800 |
| Xiu | 1800 |

total_input = 5400，difference = −400

**預期結果**：

- Ning effective_split = 1800 + (−400) = 1400
- Ning 應收金額相對縮小（等於少分攤 $400）
- Σcost = 1400 + 1800 + 1800 = 5000 ✅（與 twd_amount 對齊）

---

### TC-DIFF-03：付款人未參與分帳

**場景**：Ning 付款但自己不參與（如代付全家門票，自己沒參加）

**前置資料**：

twd_amount = 2400，付款人 = Ning（is_participating = false）

| 成員 | split_amount | is_participating |
|------|-------------|-----------------|
| Ning | — | false |
| Ziyu | 800 | true |
| Xiu | 800 | true |
| Mei | 700 | true |

total_input = 2300，difference = +100

**預期結果**：

| 成員 | payout | cost | net_balance |
|------|--------|------|------------|
| Ning | 2400 | **100**（差額） | **+2300** |
| Ziyu | 0 | 800 | −800 |
| Xiu | 0 | 800 | −800 |
| Mei | 0 | 700 | −700 |

**驗算**：2300 − 800 − 800 − 700 = **0 ✅**

**結算**：3 筆，Ziyu → Ning $800、Xiu → Ning $800、Mei → Ning $700

---

> ⚠️ **v1.1 修正說明（TC-DIFF-03 文件化）**
>
> 原始文件（v1.0）載明「Σnet_balance ≠ 0 屬正常」，此為**錯誤描述**。
>
> 實際實作中，`difference = twd_amount − Σ(participants' split_amount)` **一律加入付款人 cost**，
> 無論付款人是否 `is_participating`（見 settlement-engine.ts 第 170 行）。
>
> 因此：
> - Ning cost = 0（未參與）+ 100（差額）= **100**（非 0）
> - Ning net_balance = 2400 − 100 = **+2300**（非 +2400）
> - **Σnet_balance 恆為 0**，守恆規則在任何情況下均成立。
>
> 演算法文件 §4.3 的描述「差額不影響其他人；付款人 cost 為 0」已失準，
> 應更新為：「差額歸付款人 cost（即使未參與），確保 Σnet_balance = 0」。
> → **建議 Data Analyst 更新演算法文件 §4.3**。

---

### TC-DIFF-04：差額為零（完全對齊）

## 4.6 浮點與精度

### TC-FLOAT-01：shared 均攤有餘數，付款人吸收

**場景**：3 人均攤 $1000

計算：`1000 / 3 = 333.33...` → ROUND = 333

remainder = 1000 − (333 × 3) = +1

**前置資料**：

| 成員 | 角色 |
|------|------|
| Ning | 付款人 |
| Ziyu | 參與 |
| Xiu | 參與 |

**預期結果**：

- Ziyu cost = 333
- Xiu cost = 333
- Ning cost = 333 + 1（吸收餘數）= 334
- Σcost = 333 + 333 + 334 = 1000 ✅

---

### TC-FLOAT-02：shared 均攤 ROUND 造成 −1 誤差

**場景**：3 人均攤 $100

計算：`100 / 3 = 33.33...` → ROUND = 33

remainder = 100 − (33 × 3) = +1 → 付款人吸收 +1

**預期結果**：

- 非付款人各 $33
- 付款人 $34
- Σcost = 100 ✅

---

### TC-FLOAT-03：結算轉帳金額 < 1 元，捨去

**場景**：Xiu net_balance = −0.4（浮點誤差）

**預期結果**：

- |−0.4| < 1 → 視為已平衡
- 不產生「Xiu → 任何人 $0」的 settlement_item
- Xiu 移出 debtors

---

### TC-FLOAT-04：外幣記錄，twd_amount 為整數，外幣顯示兩位小數

**場景**：Ning 在日本刷了 ¥12,340 JPY，換算 twd_amount = 2,777

**預期結果**：

- 結算計算使用 twd_amount = 2777（整數）
- 外幣欄位顯示 ¥12,340.00（2 位小數）
- exchange_rate 顯示 = 2777 / 12340 ≈ 0.2251（僅供參考）
- 不影響結算金額

---

### TC-FLOAT-05：多筆 shared 累積的最小轉帳精度

**場景**：5 筆 shared，各自 ROUND 後 Σnet_balance 有浮點誤差

**前置資料**：

| 筆 | twd_amount | 人數 | per_person |
|----|-----------|------|-----------|
| A | 1000 | 3 | 333（+1 余，Ning吸收） |
| B | 2000 | 3 | 667（−1 余，Ning吸收） |
| C | 500 | 3 | 167（−1 余，Ning吸收） |

**預期結果**：

- 每筆餘數個別計算並歸付款人（非累積再處理）
- Σnet_balance 的浮點誤差若 < 1 元，在 settlement 階段捨去
- 不允許產生 $0 的 settlement_item

## 4.7 重複結算（結算後修改）

### TC-RESET-01：結算後修改帳目金額，觸發重算

**流程**：

1. 初始結算 S1：Ziyu → Ning $2700，Mei → Ning $1900，Xiu → Ning $100
2. 修改筆 A（午餐）twd_amount：1200 → 2000
3. **預期觸發**：trip.status → active，S1 status → superseded

**執行**：重新結算 → S2

**預期結果**：

- S2 以修改後金額重新計算
- S1 記錄保留（歷史備查），不影響 S2
- 前端顯示最新 S2 的 settlement_items
- S1 標記為「歷史版本」可展開查看

---

### TC-RESET-02：結算後新增帳目

**流程**：

1. S1 已 confirmed
2. 新增一筆帳目 D（shared $600，全員）
3. trip.status → active

**執行**：重新結算 → S2

**預期結果**：

- S2 包含 D 的計算
- D 的每人 cost += 150
- settlement_items 更新（不是在 S1 基礎上增量，而是**全量重算**）

---

### TC-RESET-03：重複結算後 is_cleared 狀態不污染新結算

**場景**：S1 中 Ziyu 已在 app 外自行轉帳，標記 is_cleared = true；重算 S2 後，Ziyu 在 S2 中的轉帳應重新從未清算狀態開始

**預期結果**：

- S2 的 settlement_items 全部 is_cleared = false（初始狀態）
- S1 的 is_cleared 標記不繼承到 S2
- 使用者需在 S2 中重新確認清算

### TC-COMBO-02：Σnet_balance 必為零（守恆驗證）

**驗證方式**：對每個結算結果執行：

## §5 trip.status 狀態情境測試

> 對應資料模型 §7「trip.status 四個值定義與狀態切換機制」  
> 測試腳本：`supabase/functions/__tests__/trip-status.test.ts`（19 個 TC，全數通過）

### 架構說明

trip.status 分兩層驗證：

**層一：前端派生（`deriveDisplayStatus` 純函式）**

```ts
if (status === 'settled')  → 'settled'
if (status === 'archived') → 'archived'
today < start_date         → 'planned'
today >= start_date        → 'active'
```

`planned` / `active` 的顯示不依賴 DB 欄位，由日期即時計算。

**層二：DB 寫入規則（三個 Edge Function）**

| 操作 | DB status 寫入值 |
|------|----------------|
| 建立行程 | `planned` |
| `confirm-settlement` | `settled` |
| 使用者封存 | `archived` |
| `reopen-settlement`（mode=reopen，today < start_date） | `planned` |
| `reopen-settlement`（mode=reopen，today ≥ start_date） | `active` |
| `reopen-settlement`（mode=unarchive） | `settled` |

---

### TC-STATUS-01：建立行程，今天在出發前，顯示 planned

**場景**：today = 2026-07-20，start_date = 2026-08-01，DB status = planned

**預期結果**：`deriveDisplayStatus` → `planned`

---

### TC-STATUS-02：今天已到出發日，DB 仍是 planned（舊值），前端派生 active

**場景**：today = 2026-07-01（≥ start_date），DB status = planned（未更新）

**預期結果**：`deriveDisplayStatus` → `active`（不信任 DB planned 值）

> 核心設計驗證：前端永遠以日期為準，DB 的 planned/active 值只是初始寫入，不保證即時準確。

---

### TC-STATUS-03：出發日前一天 vs 當天的精確邊界

| today | start_date | 預期顯示 |
|-------|-----------|---------|
| 2026-07-31 | 2026-08-01 | `planned` |
| 2026-08-01 | 2026-08-01 | `active`（出發當天算 active） |
