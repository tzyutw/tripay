## 關聯關係

```
profiles        1 ──＜  trips               （一個帳號多趟行程）
trips           1 ──＜  trip_members        （行程成員）
trips           1 ──＜  expenses            （消費紀錄）
trips           1 ──＜  settlements         （可多次結算）
expenses        1 ──＜  expense_splits      （分帳明細）
trip_members    1 ──＜  expense_splits      （參與分帳）
settlements     1 ──＜  settlement_items    （每對成員應付/應收）
trip_members    1 ──＜  settlement_items    （from_member_id / to_member_id 雙向）
```

## 設計說明

1. **待填狀態**：`foreign_amount` / `twd_amount` 允許 NULL，搭配獨立 bool flag（`foreign_pending` / `twd_pending`），兩個維度可各自待填，互不影響。

2. **結算後可修改**：settlements 不做唯一約束，重新計算新增一筆即可。`reopen-settlement` 將 `trip.status` 回退為 `active`（行程日期仍在進行中）或 `planned`（尚未出發），依 `today` vs `start_date` 判斷。

3. **唯讀分享**：`trips.share_token` 搭配 Supabase RLS policy，持 token 者唯讀全行程資料。

4. **分帳邏輯**：
   - `shared`：expense_splits 的 split_amount 為 NULL，由演算法均攤或按比例計算
   - `individual`：逐一輸入每位成員 split_amount
   - `personal`：不產生 expense_splits，不進分帳

5. **Phase 1.5 啟用條件**：開啟 `trips.collab_enabled`，`trip_members.linked_profile_id` 連結真實帳號，新增 `trip_collaborators` 管理權限，主表結構不需更動。

6. **Phase 2 啟用條件**：`expenses.card_id` 連結 `cards` 表，啟用副卡記帳。

7. **trip.status 四個值定義與狀態切換機制**（決策 A，2026-06-26）

   | 值 | 語意 | 前端顯示文案 | 觸發條件 |
   |----|------|------------|---------|
   | `planned` | 行程已建立，尚未出發 | 準備旅遊 | 建立行程時寫入 |
   | `active` | 旅遊進行中 | 旅遊中 | 使用者操作或前端派生 |
   | `settled` | 已執行結算 | 已結算 | 呼叫 `confirm-settlement` 時寫入 |
   | `archived` | 封存（預設只讀） | 封存 | 使用者主動封存 |

   **狀態切換機制選型：前端派生（推薦）**

   `planned` / `active` 的顯示由**前端依日期即時計算**，不依賴 DB 欄位：

   ```ts
   function deriveDisplayStatus(trip: Trip, today: Date): DisplayStatus {
     if (trip.status === 'settled')  return 'settled';
     if (trip.status === 'archived') return 'archived';
     // planned / active 依日期派生
     return today < new Date(trip.start_date) ? 'planned' : 'active';
   }
   ```

   **不採用 cron/trigger 自動更新 DB** 的理由：
   - Phase 1 成員數少、行程數少，前端計算成本可忽略
   - 避免 pg_cron 設定複雜度與潛在時區問題
   - DB 的 `planned`/`active` 值僅作為初始寫入與未來伺服器端篩選用途，不保證即時準確
   - 真正需要精確狀態的操作（結算、封存）都是使用者主動觸發，不依賴自動切換

   **DB 寫入規則（簡化）：**
   - 建立行程：寫 `planned`
   - 呼叫 `confirm-settlement`：寫 `settled`
   - 使用者封存：寫 `archived`
   - 重新開啟（`reopen-settlement`）：寫 `planned` 或 `active`（依 `today` vs `start_date` 判斷）
   - 解除封存（`unarchive-trip`）：寫 `settled`（見 §8）

8. **解除封存（決策 B，2026-06-26）**

   `archived → settled`，不回退 `active`/`planned`（封存的行程通常已結束旅遊，回到結算狀態最合理）。

   **不需新增 Edge Function**，在現有 `reopen-settlement` 中擴充一個 `unarchive` 操作即可：

   ```
   POST /functions/v1/reopen-settlement
   Body: { trip_id, mode: "unarchive" }  ← 新增 mode 參數
     mode = "reopen"   → settled → active/planned（修改帳目重算）
     mode = "unarchive" → archived → settled（解除封存，恢復可讀寫）
   ```

   前端入口：封存行程的操作選單加「重新開啟行程」按鈕，呼叫 `mode: "unarchive"`。
