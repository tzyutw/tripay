## 0. 術語定義

| 術語 | 定義 |
|------|------|
| **付款人**（payer） | `expenses.payer_member_id` 指定的成員，即實際掏錢的人 |
| **參與成員**（participants） | `expense_splits.is_participating = true` 的成員集合 |
| **split_amount** | `expense_splits.split_amount`，individual 模式下各人輸入金額；shared 模式為 NULL |
| **twd_amount** | `expenses.twd_amount`，該筆消費的台幣總金額 |
| **twd_pending** | `expenses.twd_pending = true`，表示台幣金額尚未填寫 |
| **split_pending** | `expense_splits.split_pending = true`，表示 individual 模式下該成員金額尚未填寫 |
| **淨餘額**（net_balance） | 某成員「實際付出」−「應分攤總額」 |

## 1. 各費用類型的分攤計算

### 1.1 三種費用類型概覽

| expense_type | 分攤邏輯 | expense_splits 寫入 |
|-------------|----------|-------------------|
| `shared` | 均攤：台幣總額 ÷ 參與人數 | `split_amount = NULL`（由演算法算） |
| `individual` | 各自輸入金額 | `split_amount` 逐人填入，待填者為 NULL + `split_pending = true` |
| `personal` | 不進分帳，僅記錄 | 無 expense_splits 記錄 |

---

### 1.2 shared 模式：均攤計算

```
per_person_amount = ROUND(twd_amount / count(participants), 0)
```

**精度規則**：台幣取整數（`ROUND(..., 0)`）。

**餘數處理**：由於 `ROUND` 可能造成各人加總與總額相差 ±1 元，依以下規則處理：

```
remainder = twd_amount − (per_person_amount × count(participants))

若 remainder ≠ 0：
  → 將差額加到「付款人」的分攤金額上
  → 即付款人吸收捨入誤差
```

> **理由**：家庭旅遊場景下，1 元誤差通常由付款人（通常是 Ning🍋）吸收是最自然的行為，避免四捨五入爭議。

---

### 1.3 individual 模式：各自金額計算

individual 模式下，每位參與成員各自輸入自己的 `split_amount`。

```
各人應分攤額 = split_amount（直接使用，無需計算）
```

**差額處理（Phase 1 預設行為）**：

```
total_input = Σ(split_amount) for all participants where split_amount IS NOT NULL

difference = twd_amount − total_input
```

| 情況 | 說明 | Phase 1 處理 |
|------|------|------------|
| `difference = 0` | 完全對齊 | 正常 |
| `difference ≠ 0` | 模糊分帳（如免稅湊單） | **差額歸付款人**（付款人自行吸收） |
| `difference > 0` | 各人填的加總少於實付 | 付款人多付這個差額，視為付款人自吸 |
| `difference < 0` | 各人填的加總超過實付 | 付款人相對少付，差額從付款人應收中扣除 |

**實作方式**：計算付款人的有效分攤額時，加入差額修正：

```
payer_effective_split = payer.split_amount + difference
```

若付款人未參與分帳（`is_participating = false`），差額計入付款人的 **effective cost**（而非 payout）：

```
payer.effective_cost = difference
payer.net_balance    = twd_amount（payout）− difference（effective_cost）
                     = Σ(participants' split_amount)
```

此設計確保 **Σnet_balance 恆為 0**（TC-DIFF-03 驗證）：付款人的淨餘額精確等於所有參與者 split_amount 的加總，其他人的計算不受影響。

> **前端提示行為**：輸入時即時顯示「已填 ¥X / 總額 ¥Y」；差額 ≠ 0 時顯示橘色差額警示，但**允許儲存**（不阻斷流程）。

---

### 1.4 personal 模式

`personal` 費用**完全不進分帳計算**：
- 不建立 `expense_splits` 記錄
- 不計入任何成員的應分攤額
- 不計入「全行程總支出」統計
- 記帳人自行負擔，不影響結算

## 2. 「我的花費」計算公式

> **定義**：`owner_member_id` 對應成員在整趟行程中的**應分攤總額**（不是「付了多少錢」，而是「該負擔多少錢」）。

### 2.1 完整計算公式

```
my_cost = Σ(my_share_per_expense) for all expenses where NOT twd_pending

my_share_per_expense 依費用類型計算如下：
```

| expense_type | 計算方式 | 條件 |
|-------------|----------|------|
| `shared` | `ROUND(twd_amount / count(participants), 0)` + 餘數（若我是付款人） | `is_participating = true` |
| `shared` | `0` | `is_participating = false` |
| `individual` | `split_amount`（我的欄位） | `is_participating = true` 且 `split_pending = false` |
| `individual` | `0`（排除，不計） | `split_pending = true` |
| `personal` | `0`（不進分帳） | — |

### 2.2 待填筆在「我的花費」中的處理

```
若 twd_pending = true：整筆排除，不計入 my_cost
若 split_pending = true（individual 模式）：該成員的 split_amount 視為 0，不計入其 my_cost
```

**前端顯示規則**（方案 C，已在旅程地圖中確認為推薦方案）：
- 統計卡顯示「我的花費 $X」
- 旁邊附加「⚠️ 含 N 筆待填，數字僅供參考」
- N = 影響我的分攤計算的待填筆數（`twd_pending = true` 且我有參與，或 `split_pending = true` 且我的那份待填）

### 2.3 「我的花費」vs「我的付出」區分

| 名稱 | 定義 | 用途 |
|------|------|------|
| **我的花費**（my_cost） | 我應分攤的金額總和 | 即時統計卡、個人視角切換 |
| **我的付出**（my_payout） | 我作為付款人實際付出的金額總和 | 結算頁「計算依據」展示 |

```
my_payout = Σ(twd_amount) for all expenses where payer_member_id = me AND NOT twd_pending
```

## 3. 結算演算法：Minimum Transactions

### 3.1 演算法目標

最小化轉帳筆數，讓每個人只需轉帳給必要的對象，而非所有人互相轉帳。

### 3.2 Step 1：計算每人淨餘額

```
對每位成員 M：

  payout(M) = Σ(twd_amount) for all expenses where:
    - payer_member_id = M
    - twd_pending = false

  cost(M) = Σ(my_share_per_expense) for all expenses where:
    - M is participating
    - twd_pending = false
    - （individual 模式下：split_pending = false）

  net_balance(M) = payout(M) − cost(M)
```

| net_balance | 意涵 |
|------------|------|
| `> 0` | M 付多了，應**收回**這個金額 |
| `< 0` | M 付少了，應**支出**這個金額 |
| `= 0` | M 已平衡 |

### 3.3 Step 2：Minimum Transactions 匹配

使用貪婪演算法（Greedy Approach）：

```
creditors = 所有 net_balance > 0 的成員，按金額降序排列
debtors   = 所有 net_balance < 0 的成員，按絕對值降序排列

while creditors 不空 且 debtors 不空：
  creditor = creditors[0]    // 應收最多的人
  debtor   = debtors[0]      // 應付最多的人

  transfer = min(creditor.balance, abs(debtor.balance))

  → 記錄一筆：debtor 轉給 creditor，金額 = ROUND(transfer, 0)

  creditor.balance -= transfer
  debtor.balance   += transfer

  若 creditor.balance ≈ 0（< 1 元）：移出 creditors
  若 debtor.balance   ≈ 0（> −1 元）：移出 debtors
```

**輸出**：一組 `settlement_items`，每筆格式為：
```
{ from_member_id, to_member_id, amount }
```

### 3.4 精度收斂規則

```
浮點誤差收斂閾值：絕對值 < 1 元 → 視為 0，捨去不記錄轉帳
最終轉帳金額：ROUND(amount, 0)，台幣以元為單位
```

> 確保所有轉帳金額為整數，避免「應付 $301.5」的尷尬。

## 4. 邊界條件完整列表

### 4.1 待填金額（twd_pending）

| 情況 | 結算頁行為 | settlement_items |
|------|----------|-----------------|
| 有 1+ 筆 `twd_pending = true` | 結算按鈕可點擊，但顯示⚠️警告：「仍有 N 筆台幣待填，結算結果可能不完整」 | 待填筆完全排除在計算之外，不計入任何人的 payout / cost |
| 全部補齊後重算 | 結算後可重新計算（新增一筆 settlement，trip.status 回退 active） | 重算後自動覆蓋舊 settlement_items |

### 4.2 split_pending（individual 模式部分成員待填）

| 情況 | 對應處理 |
|------|---------|
| A 的 `split_pending = true` | A 的 split_amount 視為 0，不計入 A 的 cost；差額全部歸付款人吸收 |
| 全員皆 `split_pending = true` | 整筆 individual 費用的 cost 全為 0；twd_amount 完全由付款人的 payout 承擔（付款人淨餘額偏高，待補填後重算） |
| 重新補填後 | 更新 split_amount，清除 split_pending；觸發重新結算 |

### 4.3 individual 模式差額歸付款人

已在 §1.3 說明，補充邊界：

| 邊界情況 | 處理方式 |
|---------|---------|
| 付款人本身未參與分帳（`is_participating = false`） | 差額計入付款人的 effective_cost（非 0）；`net_balance = twd_amount − difference = Σ(participants' split_amount)`，確保 Σnet_balance = 0（TC-DIFF-03 驗證）|
| 差額為負值（各人加總超過實付） | 付款人 effective_split 減少，結算時付款人應收金額相對縮小 |

### 4.4 personal 費用

| 情況 | 處理 |
|------|------|
| 記帳者標記某筆為 personal | 無 expense_splits，不計入任何人 payout / cost |
| 付款人付了 personal 費用 | payout 不包含此筆（個人自付，不進結算） |

### 4.5 零參與人

| 情況 | 處理 |
|------|------|
| 某筆 shared 費用無任何人 is_participating = true | 視為資料錯誤；前端應在儲存時驗證至少一位成員參與 |
| 行程只有一人 | 結算無意義；net_balance 必然為 0，不產生 settlement_items |

### 4.6 浮點與精度

| 規則 | 說明 |
|------|------|
| 台幣 | 取整數（ROUND to 0 decimal places） |
| 外幣 | 保留兩位小數（ROUND to 2 decimal places），僅用於顯示，不參與結算計算 |
| 匯率 | 回推公式：`exchange_rate = twd_amount / foreign_amount`，僅供參考顯示 |
| 結算金額 < 1 元 | 捨去（視為已平衡），不產生 settlement_item |

### 4.7 重複結算（結算後修改）

```
流程：
1. 使用者修改某筆帳目（金額 / 費用類型 / 分攤成員）
2. trip.status → active（自動回退）
3. 新增一筆 settlements 記錄（status = draft）
4. 重新執行 §3 演算法，產生新的 settlement_items
5. 使用者確認後 status → confirmed
6. 舊 settlement 記錄保留（歷史備查），is_cleared 標記保留但不影響新結算
```

## 5. 演算法完整流程圖（文字版）

```
[觸發結算]
    ↓
[Step 1] 掃描所有 expenses（trip_id = 本行程）
    ├─ 排除 twd_pending = true 的筆
    ├─ 排除 expense_type = personal 的筆
    └─ 剩下的進入計算
    ↓
[Step 2] 計算每人 payout
    └─ 付款人的 twd_amount 加總
    ↓
[Step 3] 計算每人 cost（依 expense_type）
    ├─ shared → ROUND(twd_amount / participants, 0)，餘數歸付款人
    └─ individual → split_amount（排除 split_pending = true 的項目）
                    差額（difference）歸付款人的 effective_split
    ↓
[Step 4] 計算 net_balance = payout − cost
    ↓
[Step 5] Minimum Transactions 貪婪匹配
    └─ 輸出 settlement_items { from, to, amount }
    ↓
[Step 6] 寫入 settlements + settlement_items
    └─ status = draft → 使用者確認 → confirmed
```

## 6. 計算範例：北海道四寶團

**成員**：Ning🍋、Ziyu🐟、Xiu🐵、Mei🐱

**帳目**：

| 筆次 | 類型 | 台幣金額 | 付款人 | 參與成員 | 各人分攤 |
|------|------|---------|--------|---------|---------|
| 午餐 | shared | $1,200 | Ning | 全員 | 各 $300 |
| Uniqlo | individual | $7,000 | Ning | 全員 | Ning $2,500、Ziyu $1,800、Xiu $1,600、Mei $1,000（加總 $6,900，差額 $100 歸 Ning） |
| 保險 | personal | $800 | Ziyu | — | 不進分帳 |
| 門票 | shared | $2,400 | Xiu | 全員 | 各 $600 |

**Step 2 — 每人 payout**：

| 成員 | payout |
|------|--------|
| Ning | $1,200 + $7,000 = $8,200 |
| Ziyu | $0（付保險 $800 為 personal，不算） |
| Xiu | $2,400 |
| Mei | $0 |

**Step 3 — 每人 cost**：

| 成員 | 午餐 | Uniqlo（effective） | 門票 | cost 合計 |
|------|------|---------------------|------|---------|
| Ning | $300 | $2,500 + $100（差額） = $2,600 | $600 | $3,500 |
| Ziyu | $300 | $1,800 | $600 | $2,700 |
| Xiu | $300 | $1,600 | $600 | $2,500 |
| Mei | $300 | $1,000 | $600 | $1,900 |

**Step 4 — net_balance**：

| 成員 | payout | cost | net_balance |
|------|--------|------|------------|
| Ning | $8,200 | $3,500 | **+$4,700**（應收） |
| Ziyu | $0 | $2,700 | **−$2,700**（應付） |
| Xiu | $2,400 | $2,500 | **−$100**（應付） |
| Mei | $0 | $1,900 | **−$1,900**（應付） |

**驗算**：Σ net_balance = $4,700 − $2,700 − $100 − $1,900 = **$0 ✅**

**Step 5 — 最小轉帳匹配**：

```
creditors: [Ning +4700]
debtors:   [Ziyu −2700, Mei −1900, Xiu −100]

Round 1：Ziyu → Ning $2,700　（Ning 剩 +2000，Ziyu 歸零）
Round 2：Mei  → Ning $1,900　（Ning 剩 +100，Mei 歸零）
Round 3：Xiu  → Ning $100　　（全部歸零）
```

**結算結果**：
```
Ziyu🐟 付給 Ning🍋 $2,700
Mei🐱  付給 Ning🍋 $1,900
Xiu🐵  付給 Ning🍋 $100
```

## 7. 實作備注

### 7.1 計算時機

- **即時統計**（我的花費、全行程總支出）：純前端計算，每次新增 / 修改帳目後觸發，無需 API call。
- **結算演算法**：由後端執行（Supabase Edge Function），確保所有 client 看到的結算結果一致。

### 7.2 效能邊界

- Phase 1 假設單一行程帳目數 ≤ 200 筆，成員 ≤ 10 人。
- Minimum Transactions 演算法複雜度 O(n log n)，成員數在此範圍內效能無虞。
- 前端即時統計建議使用 `useMemo`，依帳目 array 作為 dependency。

### 7.3 資料一致性

- `expense_splits` 建立時機：expenses 儲存時同步建立，不允許 orphan expenses（無 splits 的 shared / individual 費用）。
- `split_amount` 的加總驗證：僅做警示，不做阻擋（允許模糊分帳）。
