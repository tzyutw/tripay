## G-04：唯讀分享連結自動生成 OG 圖片

### 目標
行程分享連結在 LINE / 社群平台貼上時，自動顯示行程預覽卡片（行程名稱、emoji、成員數、總支出等）。

### 方案比較

| 方案 | 原理 | Edge Function 相容 | 冷啟動速度 | 維護成本 | 推薦度 |
|------|------|:-----------------:|:--------:|:-------:|:-----:|
| **Satori**（推薦） | JSX → SVG → PNG，純 Wasm，無 headless browser | ✅ 完全相容 | 快（< 200ms） | 低 | ⭐⭐⭐ |
| @vercel/og | Satori 封裝，最佳化給 Vercel Edge Runtime | ⚠️ 可用但需手動 polyfill | 快 | 低 | ⭐⭐ |
| Puppeteer / Chrome | Headless browser 截圖 | ❌ 不可（記憶體超限） | 慢（2–5s） | 高 | ❌ |
| 前端靜態 meta tag | 無動態圖，用固定佔位圖 | — | — | 最低 | 僅備用 |

### 推薦方案：Satori + Supabase Edge Function

```
GET /functions/v1/og-image?token={share_token}
  → 查 trips（by share_token）
  → 產生 JSX template（行程名、emoji、成員 emoji 列、台幣總支出）
  → Satori 轉 SVG
  → resvg-wasm 轉 PNG
  → Response: image/png, Cache-Control: public, max-age=3600
```

**meta tag 掛載點**：唯讀分享頁（`/share/[token]`）的 SSR head 注入：
```html
<meta property="og:image" content="https://{project}.supabase.co/functions/v1/og-image?token={token}" />
```

### 技術成本估算

| 項目 | 評估 |
|------|------|
| Satori + resvg-wasm bundle size | ~2.5 MB（Edge Function 上限 10 MB，可行） |
| 新增 Edge Function | 1 支（`og-image`） |
| 資料模型調整 | **不需要**（`share_token` 已在 `trips` 表存在） |
| 前端調整 | 唯讀分享頁加 meta tag 注入（< 1 天） |
| 圖片 cache | `Cache-Control: public, max-age=3600`，行程更新後 1 小時內自動刷新 |
| 總工時估算 | **3–4 天**（含設計卡片 template） |

### 風險

- Satori 不支援所有 CSS 屬性（flex 支援完整，但 grid / backdrop-filter 不支援）→ template 設計需配合限制
- 行程有 emoji 時需確認 emoji font 載入（Noto Emoji 約 1 MB）

### 結論

**技術可行，推薦 Phase 1.5 採用 Satori 方案。** 資料模型不需調整，主要成本在 Edge Function 實作與 OG 卡片視覺設計。

## G-09：新建行程時預填上趟成員資料

### 目標

建立新行程時，提供「沿用上次成員」選項，預填上一趟行程的成員名稱 + emoji，減少重複輸入。

### 讀取方式

```sql
-- 查詢使用者最近一趟行程的成員
SELECT tm.name, tm.emoji, tm.sort_order
FROM trip_members tm
JOIN trips t ON tm.trip_id = t.id
WHERE t.owner_id = :user_id
  AND t.deleted_at IS NULL
ORDER BY t.created_at DESC, tm.sort_order ASC
LIMIT 20  -- 最多取 10 人（Phase 1 成員上限）
```

前端實作方式：在 S-02（建立行程頁）的成員輸入區上方加一個「沿用上次成員 ✦」按鈕，點擊後預填，使用者可逐一編輯或刪除。

### 資料模型調整評估

**不需要調整資料模型。** 現有 `trip_members`（name / emoji / sort_order）欄位已足夠，直接查最近一趟行程的成員即可。

唯一需要決策的是：「最近一趟」的定義。

| 方案 | 排序依據 | 說明 |
|------|---------|------|
| **A. 最近建立**（推薦） | `trips.created_at DESC` | 簡單直覺，預設行為 |
| B. 最近出發 | `trips.start_date DESC` | 語意更精確，但需處理未來行程 |
| C. 使用者手動選 | UI 下拉選擇沿用哪趟 | 彈性最高，但 MVP 層級過複雜 |

**推薦方案 A**：`created_at DESC`，Phase 1.5 先做最簡單的版本。

### 邊界條件

| 情境 | 處理 |
|------|------|
| 首次建立行程（無歷史） | 不顯示「沿用」按鈕 |
| 上趟只有 1 位成員 | 正常預填，使用者可繼續新增 |
| 上趟成員 emoji 與新行程重複 | 允許（emoji 僅是識別符，不做唯一驗證） |

### 技術成本估算

| 項目 | 評估 |
|------|------|
| API 查詢 | 1 支 query（如上），可整合進 S-02 初始載入 |
| 前端 UI | S-02 成員輸入區加一個 CTA 按鈕 + 預填邏輯（< 0.5 天） |
| 資料模型調整 | **不需要** |
| 總工時估算 | **0.5–1 天** |

### 結論

**技術簡單，資料模型不需調整，工時低，推薦 Phase 1.5 優先執行。**
